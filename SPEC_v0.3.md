# CEST v0.3 仕様書

## コンセプト

**制約ベースの反復的オフィス配置意思決定支援エンジン**

| | v0.1 | v0.3 |
|---|---|---|
| 目的 | 単一オフィス比較 | 多拠点配置の最適化 + 反復的意思決定支援 |
| 評価方式 | 重み付きスコア | パレートフロンティア + 制約フィルタ |
| ハザード | なし | 警告表示のみ（CESTの責務外） |
| 出力 | KPI数値 | 人間の言葉の説明パック + 地図 + 散布図 |
| 使い方 | 1回呼んで終わり | 制約を足しながら繰り返し呼ぶ |

---

## ワークフロー

```
Round 1: 制約なし
  入力: 社員の最寄り駅+部署, オフィス候補(駅, 賃料, 面積), 出社日数
  処理: 組み合わせ列挙 → 部署配置最適化 → パレート抽出
  出力: 全候補一覧（パレート最適をハイライト）+ 各案の説明 + 地図 + 散布図

Round 2: ユーザーが制約追加
  「品川は必ず使う」「開発と企画は同じ拠点」
  → 候補が絞られた新しい一覧
  → 「制約追加で候補が5件→3件に減少」

Round 3: さらに制約追加
  「営業は品川」
  → さらに絞られた一覧
  → ユーザーが納得して決定
```

## 入力

### 社員データ (home_station_distribution)

```json
[
  { "station_id": "omiya", "count": 5, "group": "開発" },
  { "station_id": "yokohama", "count": 8, "group": "営業" }
]
```

### オフィス候補 (office_candidates)

```json
{
  "office_id": "shinagawa",
  "name": "品川オフィス",
  "nearest_station_id": "shinagawa",
  "last_mile_minutes": 3.0,
  "rent_jpy_month": 3000000,
  "floor_area_sqm": 500,
  "capacity_people": null
}
```

- `floor_area_sqm`: 物件情報から転記。不動産情報には必ず載っているため、ユーザーの入力負荷が低い
- `capacity_people`: 直接わかっている場合（自社ビル、内見済み等）に入力。指定時は面積からの推定より優先
- どちらか一方を必須とする。両方ある場合は `capacity_people` を優先

#### 面積→収容人数の変換

```
estimated_capacity = floor_area_sqm / sqm_per_person
```

- `sqm_per_person`: デフォルト 3.3㎡（国交省オフィス基準）。Settings で上書き可能
- この変換はざっくりの規模感で候補を絞るためのもの。実際に席・会議室・設備が足りるかはオフィスごとに条件が異なるため、ユーザーが候補を上から順に検討して判断する前提

### 制約 (settings)

| 制約 | フィールド | 型 | 例 |
|---|---|---|---|
| このオフィスは必ず使う | `fixed_offices` | `list[str]` | `["shinagawa"]` |
| この部署はこのオフィス | `fixed_assignment` | `list[{group, office_id}]` | `[{"group":"営業","office_id":"shinagawa"}]` |
| この部署同士は同じ拠点 | `group_together` | `list[list[str]]` | `[["開発","企画"],["人事","総務"]]` |
| 予算上限 | `budget_total_rent_jpy_month` | `int?` | `5000000` |
| 通勤上限 | `max_p95_trip_minutes` | `int?` | `90` |
| 一人あたり面積 | `sqm_per_person` | `float` | `3.3`（デフォルト） |
| 拠点数 | `num_offices` | `list[int]` | `[1, 2, 3]` |
| 出社日数 | `policy_as_is.office_days_per_week` | `float` | `3.0` |

- `fixed_assignment`, `group_together` は複数入力可能（UIで行を追加する形式）
- `group_together` の1グループ内に2個以上の部署を指定可能
- 出社日数はUIでいじれる入力パラメータ（変更後に再計算）

## 処理パイプライン

```
1. 収容人数フィルタ
   floor_area_sqm / sqm_per_person（または capacity_people）で推定収容人数を算出
   配置人数 > 推定収容人数 の組み合わせを除外

2. 予算フィルタ
   合計賃料 > budget_total_rent_jpy_month の組み合わせを除外

3. 組み合わせ列挙
   num_offices ごとに全組み合わせを列挙
   fixed_offices を含まない組み合わせを除外

4. 部署配置最適化
   group_together に含まれる部署を1つの仮想グループとして統合
   fixed_assignment を尊重しつつ、残りの部署を通勤負荷最小のオフィスに配置
   （貪欲法 = 最適。通勤はグループ間で独立のため）

5. 通勤上限フィルタ
   p95_trip_minutes > max_p95_trip_minutes の組み合わせを除外

6. 全候補を通勤×コストで評価
   パレートフロンティアを抽出（パレート劣位な案にフラグ）
   ただし全候補を返す（パレート劣位もユーザーに見せる）

7. 感度分析
   結果の確実性を判定

8. explain 生成
   技術的KPIを人間の言葉に翻訳

9. ハザード警告
   ハザードデータがある駅は警告を Notice で通知（フィルタはしない）
```

---

## 評価方式：パレートフロンティア

重み付きスコアリングは廃止する。

**理由**: 軸が通勤と賃料の2つだけの場合、重みの設定根拠がなく偽の精密さになる。2変数のトレードオフは人間がパレートフロンティアを見れば直感的に判断できる。

- 2軸: 通勤負荷 × 月額賃料
- パレート劣位な案（通勤もコストも他の案より悪い案）にフラグを付ける
- **全候補を返す。** パレート最適な案をハイライトするが、劣位な案も表示する
- ユーザーが制約を追加して再計算 → 候補が更新される

### 通勤指標

p95だけでは通勤の全体像がわからない（全員60分の案と、大半20分で一部だけ60分の案が同じp95になる）。

以下を全て出力する：

| 指標 | 意味 |
|---|---|
| `avg_trip_minutes` | 平均通勤時間（全体の負荷感） |
| `p95_trip_minutes` | 最長通勤時間（上位5%） |
| `exceed_60_count` | 60分超の人数 |
| `exceed_90_count` | 90分超の人数 |
| `distribution` | 通勤時間の分布（30分未満/30-60分/60-90分/90分超の人数） |

### パレート軸の選択

UIでパレート軸を切り替え可能にする：

```
パレート軸: [最長通勤(p95) ▼] × 月額賃料
           [平均通勤     ▼] × 月額賃料
```

デフォルトは p95。ユーザーが「平均で見たい」と思えば切り替えられる。

## ハザード：警告のみ

**CESTの責務外。** 理由：

- 浸水以外にも地震・液状化・土砂災害など無限にある
- データの信頼性・更新頻度の責任をCESTが負うことになる
- ユーザーは自治体のハザードマップを見て自分で判断できる
- 候補に入れるかどうかはユーザーの判断

**対応方針:**

- ハザードデータがある駅には Notice で「⚠ 浸水想定区域（0.8m）」等の警告を出す
- フィルタはしない。候補リストからの除外はユーザーが手動で行う
- `hazard_thresholds` による自動フィルタは廃止

## 感度分析

**目的: 「この結果を信頼していいか」を判定する。**

揺らしたときに結果（パレートフロンティアの順位）が変わるかどうかが知りたい情報。「何分変わりました」は知りたい情報ではない。

### 揺らすパラメータ

| パラメータ | 揺らし方 | 意味 |
|---|---|---|
| last_mile_minutes | ±5min | 駅〜オフィスの実距離の不確実性 |
| office_days_per_week | 3→5日, 5→3日 | 出社方針変更時にランキングが変わるか |

### 拠点消失リスクは削除

制約を変えて再計算するワークフローで対応可能。専用の感度分析は不要。

### 出力

```json
{
  "sensitivity": {
    "ranking_stable": true,
    "summary": "ラストマイルを±5分変動させてもパレートフロンティアの順位は変わりません。結果は安定しています。",
    "details": [
      {
        "parameter": "last_mile_minutes ±5min",
        "ranking_changed": false,
        "description": "全パレート最適案がフロンティア上に維持"
      },
      {
        "parameter": "出社日数 3日→5日",
        "ranking_changed": true,
        "description": "品川+大宮案がパレート劣位になり、渋谷+横浜案が優位に"
      }
    ]
  }
}
```

- `ranking_stable`: 全パラメータで順位が変わらなければ true
- 不安定な場合: 「ラストマイルの正確な値で結果が変わります。実測を推奨」
- 安定な場合: 「多少の誤差では結果は安定しています」

### 社員数の変動

感度分析ではなく**収容余裕**として各案に表示する（推定収容人数 - 配置人数）。

## explain の設計方針

技術層と説明層を分離する。

| 技術層（APIフィールド） | 説明層（explain） |
|---|---|
| `p95_trip_minutes: 72` | 「最も通勤が長い人でも72分以内」 |
| `avg_trip_minutes: 35` | 「平均通勤35分」 |
| `exceed_60_count: 12` | 「60分超えは12人」 |
| `distribution.over_90: 2` | 「90分超: 2人」 |
| `capacity_headroom: 15` | 「余裕15人。15人以上の増員で収容超過の可能性あり」 |
| `ranking_stable: false` | 「前提条件を変えると順位が入れ替わる可能性あり」 |

- APIの内部フィールド名は技術的でよい（開発者向け）
- explain はユーザー（総務・経営企画）が読む日本語
- CESTの専門用語は explain に出さない
- 推定収容人数が面積ベースの場合「※推定値」を付記

### UI仕様

#### 入力データ確認
- 現在の入力データ（社員数、部署数、オフィス候補数、出社日数）を常に表示

#### 制約パネル
- 各制約行に [✕] ボタンで削除可能

  - 1グループ内に [+ 部署追加] で部署を何個でも追加可能（4部署セットなど）

- ドロップダウンの選択肢は入力データから自動生成
- [再計算] ボタンで `POST /evaluate` を制約付きで再リクエスト

- 横軸: 月額賃料、縦軸: 通勤指標（p95 or 平均、ドロップダウンで切替）

- パレート最適な点を線で結んでフロンティアを可視化
- 点をクリック → 詳細カードと地図がその案に切り替わる

#### 全候補一覧
- **全候補を表示する。** パレート最適な案にはバッジ「★ パレート最適」

- 通勤の分布情報（30分未満○人/30-60分○人/60分超○人）を必ず表示

- 2つの案を選んで差分を表示

## スコープ外（明示的に扱わないもの）

| 項目 | 理由 |
|---|---|
| 会議室の数・広さ | ファシリティマネジメント領域。CESTは通勤×コスト最適化に特化 |
| ビル設備・築年数 | データ取得不可、定性判断 |
| ブランド・住所のステータス | 定性的 |
| 離職確率の推定 | 信頼性の高いモデルが作れない |
| ハザード評価 | CESTの専門外。警告表示のみ対応 |

CESTは「通勤負荷の定量評価と多拠点配置の最適化」に特化する。収容人数のフィルタはざっくりの規模感で候補を絞るためのものであり、実際にオフィスが要件を満たすか（席数・会議室・設備等）はユーザーが候補を上から順に検討して判断する。
