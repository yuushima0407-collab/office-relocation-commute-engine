# CEST v0.3.3 サンプルデータ

`v0.3.3-frontend.md` の **L. サンプルデータの全面刷新** で使う具体値の一覧。
そのまま `SAMPLE_CONFIGS` に流し込めるよう、駅ID・賃料・容量・部署 pool 等を確定値で記載する。

各オフィスは `(office_id, name, nearest_station_id, last_mile_minutes, rent_jpy_month, capacity_people)` のセット。
各部署は `(name, count, pool)`。pool は駅ID配列。

## サンプル選択時のフォーム自動入力対応表

| フィールド | 入力先 |
|---|---|
| officeDays | 「週何日出社」入力 |
| 最大拠点数（numOffices の最大値） | 「最大何拠点まで」入力 |
| constraints.budgetMan | 「予算上限（万円）」入力 |
| constraints.maxAvg | 「平均通勤の上限（分）」入力 |
| constraints.desiredCapacity | 「希望定員（人分）」入力 |
| constraints.maxP95 が無い | 「通勤ワースト5%の上限」は空欄（詳細設定内） |
| baseline.name | 「現オフィス名」 |
| baseline.station | 「最寄駅ID」 |
| baseline.lastMile | 「徒歩（分）」 |
| baseline.rentMan | 「月額賃料（万円）」 |
| baseline.capacity | 「収容人数」 |

baseline セクションは開いた状態にする（折りたたみを展開）。

## 確認用：このサンプルでの期待挙動

3シナリオすべてで、計算実行後に以下が観察できれば期待通り：

- パレート最適案が **3〜6件** 並ぶ（縦一列にならない）
- 各案で平均通勤と賃料のトレードオフが見える（X軸方向に散らばる）
- baseline ◆ が散布図上に表示される
- 部署別のテーブルで「●●部署は新宿に配置」「▲▲部署は東京に配置」のように **部署単位で違うオフィス**に配置される
- conflict_alerts（部署間の平均通勤の格差）が複数件出る
