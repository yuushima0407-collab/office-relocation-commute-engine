# CEST v0.3.2 Frontend

**担当**: pachi
**前提**: v0.3.2 backend がマージ済みであること
**ステータス**: 未着手

---

## 背景

v0.3.2 バックエンドで以下が変更される。フロントをこれに対応させたい。
また v0.3.1 で未実装だった「入力データで実際に計算する」導線を完成させる。

### バックエンドの変更点
- パレート判定が `(total_cost, p95)` の2軸 → `(total_rent, p95, total_capacity)` の3軸に変更
- 各案に `rent_per_capacity`（散布図の軸切替用）
- 各案に `department_breakdown[]`（部署別の通勤時間・配置先・通勤費）
- 各案に `conflict_alerts[]`（部署間の格差警告）
- トップレベルに `robustness[]`（賃料耐性 + 収容余裕。パレート最適案のみ）
- `settings.baseline` 指定時に `baseline_diagnosis` と各案の `vs_baseline`
- v0.3.1 の `sensitivity`（感度分析）は廃止 → `robustness` に置換

### v0.3.1 からの構造変更
- 感度分析タブを廃止
- 散布図は `total_rent × p95` と `rent_per_capacity × p95` の2枚を並列表示
- タブ構成: `データ入力 | 結果`（2タブに削減）

---

## 解決したい課題

1. **CSVアップロードしても計算されない** — v0.3.1 ではCSVをパース・プレビューするだけで `/evaluate` に送信していない
2. **オフィス候補を入力できない** — 社員データはCSVで入れられるが、オフィス候補がデモ固定
3. **部署別の影響がわからない** — 全社平均だけでは「営業部は改善、開発部は悪化」が見えない
4. **どの案が安定か判断できない** — 賃料が上がっても大丈夫か、増員に耐えられるかがわからない

---

## 対応してほしいこと

### 必須

**A. データ入力の完成（社員CSV + オフィス候補 → 計算実行）**

v0.3.1 では CSV アップロードはパース・プレビューのみで、`/evaluate` への送信が未実装。
また、オフィス候補の入力手段がない。以下を対応:

**A-1. オフィス候補の入力**

手入力とCSV両方に対応:

手入力フォーム（候補が少ない場合）:
- オフィス名（テキスト）
- 最寄駅（`station_master.json` からセレクト or テキスト入力）
- ラストマイル（分、数値、デフォルト5）
- 月額賃料（円、数値）
- 収容人数（人、数値）
- 「候補を追加」ボタンで複数件入力可能

CSVアップロード（候補が多い場合）:
```csv
name,nearest_station_id,last_mile_minutes,rent_jpy_month,capacity
渋谷オフィス,shibuya,5,5000000,100
新宿オフィス,shinjuku,8,4000000,80
```

**A-2. 社員データの入力（既存の改善）**

v0.3.1 の CSV アップロードはそのまま活用。変更点:
- アップロード後のプレビューに「この社員データで計算に使います」の確認表示
- 社員データも手入力フォームで追加できるようにする（任意）

**A-3. 計算実行 + 全設定の送信**

- 最低1件のオフィス候補 + 社員データが揃ったら「計算実行」ボタンを有効化
- 社員データ（CSV）+ オフィス候補 + **全設定** を組み合わせて `/evaluate` にPOST
- レスポンスを `state.data` にセットして `render()` を呼び直す
- 結果タブに自動切替
- API未接続時はボタンを無効化し「API接続が必要です」と表示

`/evaluate` に送る `settings` のフィールド一覧（v0.3.1 で UI があるものも含め、すべて送信すること）:

| フィールド | 型 | 意味 | v0.3.1 UIの状態 |
|---|---|---|---|
| `num_offices` | int[] | 拠点数パターン（例: [1,2,3]） | UI あり |
| `fixed_offices` | string[] | 必ず採用するオフィスID | UI あり |
| `fixed_assignment` | {group, office_id}[] | 部署→オフィス固定 | UI あり |
| `group_together` | string[][] | 同じ拠点に配置する部署グループ | UI あり |
| `max_p95_trip_minutes` | int? | 通勤上限（p95片道分） | UI あり |
| `max_avg_trip_minutes` | float? | 平均通勤上限 | UI 追加が必要 |
| `budget_total_rent_jpy_month` | int? | 予算上限 | UI あり |
| `sqm_per_person` | float | 1人あたり面積（デフォルト3.3） | UI 追加が必要 |
| `commute_cost_policy` | "full"/"capped"/"ignore" | 通勤費の扱い | UI 追加が必要 |
| `commute_cost_cap_jpy_month` | int? | 通勤費上限額（capped時） | UI 追加が必要 |
| `baseline` | object? | 現オフィス（Before/After用。できれば） | UI 追加が必要 |

v0.3.1 で既にUIがあるフィールドは送信を繋ぐだけ。「UI 追加が必要」のフィールドは折りたたみセクション内に配置する（A-5 参照）。

リクエスト形式:
```json
{
  "office_candidates": [...],
  "home_station_distribution": [...],
  "policy_as_is": {
    "office_days_per_week": 3
  },
  "settings": {
    "num_offices": [1, 2],
    "fixed_offices": [],
    "fixed_assignment": [],
    "group_together": [],
    "max_p95_trip_minutes": null,
    "budget_total_rent_jpy_month": null,
    "commute_cost_policy": "full",
    "commute_cost_cap_jpy_month": null
  }
}
```

**A-4. 条件を変えて再計算**

結果を見た後に入力を変えて「計算実行」を再度押せるようにする。A-3 と同じ仕組みで、入力を書き換えて同じ `/evaluate` に投げ直すだけ。

具体的な再計算の導線:
- 結果画面から入力タブに戻れること
- 入力タブで設定を変更して再度「計算実行」
- 「この候補を除外して再計算」のようなショートカットがあると便利だが、なくても成立する

> バックエンド変更不要。既にある `/evaluate` を繰り返し叩くだけ。

**A-5. init() の修正**

- API接続時: デモデータを表示しつつ「データ入力タブからお試しください」の案内を出す
- API未接続時: デモデータ表示（現状通り）

**A-5. 入力UIの構成**

v0.3.1 の既存入力はすべてそのまま維持する。新しく追加する入力（オフィス候補、baseline 等）は折りたたみセクションにして、画面が入力で埋まらないようにする。構成の詳細（レイアウト、折りたたみの挙動等）はフロント担当の判断に任せる。

**B. 部署別影響ビュー**

候補の詳細ペインに、部署別の情報を表示する。

使うフィールド:
```json
{
  "department_breakdown": [
    {
      "group": "営業部",
      "count": 30,
      "avg_trip_minutes": 42,
      "p95_trip_minutes": 68,
      "assigned_office": "渋谷オフィス",
      "commute_cost_jpy_month": 450000
    }
  ]
}
```

最低限見えるべき情報:
- 部署名、人数、平均通勤、配置先オフィス
- `baseline` がある場合は現状との差分も表示（vs_baseline と組み合わせて）

表示形式（テーブル、カード等）はフロント担当の判断に任せる。

**C. 対立ポイントの警告表示**

使うフィールド:
```json
{
  "conflict_alerts": [
    {
      "type": "department_gap",
      "message": "営業部(p95: 68分)と経理部(p95: 45分)で23分の格差があります",
      "severity": "warning"
    }
  ]
}
```

- `severity` が `"warning"` なら警告スタイルで表示
- `conflict_alerts` が空なら非表示

**D. 注意点表示（robustness）**

使うフィールド（レスポンスのトップレベル `robustness[]`、パレート最適案ごとに1エントリ）:
```json
{
  "robustness": [
    {
      "combination_id": "k2_combo_1",
      "rent_tolerance": [
        {
          "office_id": "shibuya",
          "office_name": "渋谷オフィス",
          "current_rent": 5000000,
          "max_rent_before_drop": 6200000,
          "tolerance_pct": 24.0
        }
      ],
      "capacity_headroom": {
        "total_remaining": 30,
        "bottleneck_office": "池袋オフィス",
        "bottleneck_remaining": 2,
        "per_office": [
          {
            "office_id": "shibuya",
            "office_name": "渋谷オフィス",
            "capacity": 100,
            "assigned": 72,
            "remaining": 28
          }
        ]
      }
    }
  ]
}
```

**表示の方針**:

ユーザーは非技術者（総務・経営企画）なので、「感度分析」「頑健性」「パレート脱落」等の用語は使わない。表示名は「この案の注意点」とする。

数値をそのまま出すのではなく、原因と結果をセットで自然言語にする:
- 賃料耐性: `tolerance_pct` と `office_name` から「○○オフィスは△%値上がりしても大丈夫です」「○○オフィスは△%の値上がりで他の案が有利になります」
- 収容余裕: `remaining` から「全体であと○人まで入れます」「○○オフィスはあと△人で満員です」

✅（安心）と ⚠（注意）の使い分けの閾値や、表示のレイアウト・トーンはフロント担当が調整してよい。

`tolerance_pct` が `null` の場合は「上限なし（他に競合する案がない）」を意味する。

**E. 散布図（2つ並べて表示）**

パレート判定は3軸 `(total_rent, p95_trip, total_capacity)` で行われるため、どの2軸を表示してもパレートマークは一貫する。

2つの散布図を並べて表示する:
- **図1**: X軸 `total_rent_jpy_month`（月額賃料）× Y軸 `p95_trip_minutes` — 「いくら払って通勤どれくらいか」
- **図2**: X軸 `rent_per_capacity`（1人あたり月額賃料）× Y軸 `p95_trip_minutes` — 「1人あたりのコスパはどうか」

それぞれ別の視点なので切り替えではなく並列に出す。2つの図は連動させる: 片方の図で案をクリック（選択）したら、もう片方の図でも同じ案がハイライトされるようにする。2つの図を見比べて同じ案がどう位置づけられるか一目でわかるようにする。

3軸目の `total_capacity` はドットのサイズや色、ホバー情報で見せる等の方法が考えられる。レイアウト（横並び・縦並び等）はフロント担当の判断に任せる。

### できれば

**F. Before/After 比較（現オフィスとの差分）**

データ入力のどこかに「現在のオフィス（任意）」入力を設置:
- オフィス名、最寄駅、ラストマイル、月額賃料、収容人数

入力された場合、`/evaluate` リクエストの `settings.baseline` に含める。
未入力なら `baseline` を送らない（v0.3.1 互換動作）。

`baseline_diagnosis` がレスポンスに含まれる場合、結果画面に「現状診断」を表示:
- 現オフィス名、収容率、平均通勤、p95通勤、90分超の人数
- `alerts` を警告として表示
- `baseline_diagnosis` がない場合は非表示

各案の `vs_baseline`:
- 候補の詳細に「現状との比較」を追加
- 改善（マイナス）と悪化（プラス）が区別できるようにする
- 「60人が改善、35人が悪化」のような要約

> baseline は任意入力。まず候補比較ができることが最優先。

**G. 意思決定レポート（PDF）**

現在表示中の結果をPDFとして書き出す機能。

- html2pdf.js 等を使ってフロントのみで実装
- 含める情報: baseline_diagnosis（あれば）、パレート最適案の一覧、部署別影響、比較表
- バックエンド変更不要

---

## 注意事項

- v0.3.1 の感度分析UI（タブ含む）は削除してよい
- `robustness` はパレート最適案のみに存在する。非パレート案には対応するエントリがない
- `baseline_diagnosis` と `vs_baseline` は `baseline` 未指定時には返ってこない。関連UIを非表示にすること
- `conflict_alerts` は空配列の場合あり。空なら非表示
- v0.3.1 の既存入力項目はすべて維持すること
