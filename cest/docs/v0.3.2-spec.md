## 概要

v0.3.1 の「候補を絞る」ツールから、**「意思決定を支援する」ツール**への進化。
パレートで残った案を、関係者が納得して選べるようにする。

## v0.3.1 からの主な変更

### パレ���ト判定の変更

v0.3.1: `(total_cost_jpy_month, p95_trip_minutes)` の2軸
v0.3.2: **`(total_rent, p95_trip_minutes, total_capacity)` の3軸**

- total_rent: 低いほ��良い（会社の支出）
- p95_trip_minutes: ���いほど良い（通勤の最悪ケース）
- total_capacity: 高いほど良い（収容可能人数＝将来の増員余地）
- 2軸だと「収容人数が多いのにコストと通勤で負けているだけで脱落する」問題があったため3軸に変更
- 3軸パレートにすることで、どの2軸を散布図に表示してもパレート判定が一貫する

散布図は2つ並べて表示する: X軸 `total_rent` の図と X軸 `rent_per_capacity` の図。それぞれ「総額」と「コスパ」の異なる視点で比較できる。`rent_per_capacity` は各案の返り値に含まれる。

### 感度分析の廃止 → 注意点分析（robustness）

v0.3.1 の感��分析（last_mile ±5min / 出社日数変更）は全案に同じ影響が出るため順位が変わらず、意思決定に寄与しない。v0.3.2-spec(旧) の close_pairs（僅差ペア検出）もパレート散布図を見れば明らかなため不要。

代わりに **robustness（注意点分析）** を導入:

- **賃料耐性**: 各オフィスの賃料が何%上がるとパレートから脱落するか逆算
- **収容余裕**: オフィスごとの残り人数、ボトル��ック（最も余裕が少ないオフィス）の検出

フロントでの表示名は「この案の注意点」とし、「感度分析」「頑健性��「パレート脱落」等の専門用語は使わない。v0.4 では AI がこのデータを使って対話的に質問に答える。

### コスト↔通勤スライダーの削除

パレート最適案が少数（3〜5案）なら直接見比べた方が早い。スライダーによるスコアリングは導入しない。

## Backend 変更

### 1. パレート判定の3軸化

`mark_pareto_frontier` の支配判定を `(total_rent, p95_trip_minutes, total_capacity)` の3軸に変更。

`evaluate_combo` の返り値に `rent_per_capacity` を追加（散布図の軸切替用）:
```json
{
  "rent_per_capacity": 50000
}
```

### 2. 部署別影響ビュー

`all_combinations[]` の各案に `department_breakdown` と `conflict_alerts` を追加:

```json
{
  "department_breakdown": [
    {
      "group": "営業部",
      "count": 30,
      "avg_trip_minutes": 42,
      "p95_trip_minutes": 68,
      "assigned_office": "渋谷オフィス",
      "commute_cost_jpy_month": 450000
    }
  ],
  "conflict_alerts": [
    {
      "type": "department_gap",
      "message": "営業部(p95: 68分)と経理部(p95: 45分)で23分の格差があります",
      "severity": "warning"
    }
  ]
}
```

部署間 p95 の差が 15 分以上で `conflict_alerts` を生成。

### 3. 注意点分析（robustness）

レスポンスのトップレベルに `robustness` ���追加（パレート最適案ごとに1エントリ）:

```json
{
  "robustness": [
    {
      "combination_id": "k2_combo_1",
      "rent_tolerance": [
        {
          "office_id": "shibuya",
          "office_name": "渋谷オフィス",
          "current_rent": 5000000,
          "max_rent_before_drop": 6200000,
          "tolerance_pct": 24.0
        }
      ],
      "capacity_headroom": {
        "total_remaining": 30,
        "bottleneck_office": "池袋オフィス",
        "bottleneck_remaining": 2,
        "per_office": [
          {
            "office_id": "shibuya",
            "office_name": "渋谷オフィス",
            "capacity": 100,
            "assigned": 72,
            "remaining": 28
          }
        ]
      }
    }
  ]
}
```

**賃料耐性の算出ロジック**:
1. パレート最適案 X に対し、X を支配しうる他の案（p95 ≤ X の p95 かつ capacity ≥ X の capacity）を列挙
2. その中で total_rent が最も低い案を特定
3. X の total_rent がその案を超えるとパレートから脱落する（3軸すべてで負ける）
4. 各オフィスごとに「この額まで値上がりしても大丈夫」という閾値と変化率(%)を返す
5. 支配しうる案が存在しない場合は `max_rent_before_drop: null, tolerance_pct: null`（上限なし）

**収容余裕**: `capacity - assigned_population` をオフィスごとに返し、最も余裕が少ないオ��ィスをボトルネックとして報告。

### 4. Before/After 比較 + 現状診断

`settings.baseline` を指定すると、レスポンスに以下が追加される:

**`baseline_diagnosis`**（トップレベル）:
```json
{
  "baseline_diagnosis": {
    "office_name": "現オフィス（丸の内）",
    "employee_count": 100,
    "capacity_people": 90,
    "occupancy_pct": 111.1,
    "avg_trip_minutes": 52.3,
    "p95_trip_minutes": 87.0,
    "over_60min_count": 28,
    "over_60min_pct": 28.0,
    "over_90min_count": 6,
    "over_90min_pct": 6.0,
    "total_commute_cost_jpy_month": 480000,
    "rent_per_person": 80000,
    "alerts": [
      "収容率が100%を超えています（111%）",
      "通勤90分超の社員が6人（6%）います"
    ]
  }
}
```

**`vs_baseline`**（各案に付与）:
```json
{
  "vs_baseline": {
    "avg_trip_change": 5.2,
    "p95_trip_change": -3.0,
    "rent_change": -2000000,
    "commute_cost_change": 150000,
    "total_cost_change": -1850000,
    "worse_count": 35,
    "better_count": 60,
    "unchanged_count": 10
  }
}
```

`baseline` 未指定時は `baseline_diagnosis: null`、`vs_baseline` は各案に付与されない（v0.3.1 互換）��

## Frontend 変更

> フロントの詳細仕様は `v0.3.2-frontend.md` を参照。以下は概要のみ。

### 必須
- **データ入力の完成**: オフィス候補入力（手入力+CSV）+ 社員CSV → `/evaluate` 連携
- **部署別影響ビュー**: `department_breakdown` を候補詳細ペインに表示
- **対立ポイント警告**: `conflict_alerts` を notice スタイルで表示
- **注意点表示**: `robustness` の賃料耐性・収容余裕を自然言語で表示
- **散布図2枚**: `total_rent × p95` と `rent_per_capacity × p95` を並列表示

### できれば
- **Before/After 比較**: baseline 入力（任意）→ `baseline_diagnosis` + `vs_baseline` 表示
- **What-If シナリオ**: 入力を加工して `/evaluate` を再リク���スト（部署人数増減、候補除外）
- **意思決定レポート PDF**: html2pdf.js でフロントのみ生成
