# CEST v0.4 仕様

**ステータス**: ドラフト（v0.3.2 完了後に着手）
**前提**: v0.3.2 の必須機能（データ入力の完成、部署別影響、対立ポイント警告、散布図改善）が完成していること

---

## 概要

「候補を比較する」ツールから、**「最適なエリアを探索する」ツール**への進化。
社員データを入れるだけで、オフィスを置くべきエリアを逆算・提案する。
加えて Bedrock（Claude）による自然言語での分析支援を統合する。

---

## ユーザーストーリー

### 探索モード
> 総務担当者として、社員の最寄駅データと予算だけ入力すれば、
> どのエリアにオフィスを置くべきか候補が出てくるようにしたい。
> 候補を自分で探す手間をなくしたい。

### AI分析
> 経営層への説明資料を作るとき、分析結果について自由に質問して、
> データに基づいた回答を得たい。定型文ではなく、質問に合わせた説明がほしい。

---

## 新機能一覧

| # | 機能 | 目的 | 担当 |
|---|------|------|------|
| 1 | 探索モード（/explore） | オフィス候補エリアの自動提案 | backend + frontend |

## 1. 探索モード

### エンドポイント

```
POST /explore
```

### リクエスト

```json
{
  "employees": [
    {
      "id": "emp_001",
      "name": "田中太郎",
      "group": "営業部",
      "nearest_station_id": "musashi-kosugi",
      "days_per_week": 3
    }
  ],
  "settings": {
    "budget_per_tsubo_month": 20000,
    "required_area_tsubo": 50,
    "max_results": 20,
    "baseline": {
      "office_id": "current_office",
      "name": "現オフィス（丸の内）",
      "nearest_station_id": "tokyo",
      "last_mile_minutes": 5,
      "rent_jpy_month": 8000000
    }
  }
}
```

> `employees` は個人単位のリスト。`/evaluate` の `home_station_distribution`（駅×部署の集約形式）とは異なるが、フロントで相互変換できる。
> `baseline` を指定すると、各候補エリアと現オフィスの比較（vs_baseline）も出力する。

### 処理フロー

```
1. 駅マスタから全候補駅を取得（tokyo_core_v1.json）
2. 各駅について:
   a. 全社員からの通勤時間を推定（既存ロジック流用）
   b. エリア賃料相場テーブルからコストを推定
   c. スコアリング（通勤指標 + コスト指標）
3. 予算フィルタ（budget_per_tsubo_month 超えを除外）
4. パレートフロンティア抽出（p95_trip_minutes × estimated_rent の2軸）
5. baseline 指定時は vs_baseline を算出
6. 上位 max_results 件を返却
```

### レスポンス

```json
{
  "explore_results": [
    {
      "rank": 1,
      "station_id": "shinjuku",
      "station_name": "新宿",
      "area_rank": "A",
      "estimated_rent_per_tsubo": 25000,
      "estimated_rent_total": 1250000,
      "avg_trip_minutes": 38.5,
      "p95_trip_minutes": 62.0,
      "median_trip_minutes": 35.0,
      "within_60min_count": 85,
      "within_60min_pct": 85.0,
      "over_90min_count": 2,
      "over_90min_pct": 2.0,
      "total_commute_cost_jpy_month": 320000,
      "is_pareto_optimal": true,
      "vs_baseline": {
        "avg_trip_change": -13.8,
        "p95_trip_change": -25.0,
        "rent_change": -6750000,
        "over_90min_change": -4,
        "summary": "平均通勤14分短縮、賃料675万円削減"
      },
      "department_summary": [
        { "group": "営業部", "avg_trip_minutes": 42.0, "change_from_baseline": -8.0 },
        { "group": "開発部", "avg_trip_minutes": 35.0, "change_from_baseline": -18.0 }
      ]
    }
  ],
  "pareto_station_ids": ["shinjuku", "ikebukuro", "nakano"],
  "baseline_diagnosis": {
    "avg_trip_minutes": 52.3,
    "over_90min_count": 6,
    "over_90min_pct": 6.0,
    "alerts": ["通勤90分超の社員が6人（6%）います"]
  },
  "metadata": {
    "total_stations_evaluated": 247,
    "after_budget_filter": 180,
    "pareto_optimal_count": 8,
    "computation_time_ms": 1200
  }
}
```

### エリア賃料相場テーブル

駅マスタ（`tokyo_core_v1.json`）に `area_rank` フィールドを追加するか、別ファイルで管理:

```python
# cest/src/cest/data/area_rent.py
AREA_RENT_PER_TSUBO = {
    "S": 35000,   # 大手町・丸の内・日本橋
    "A": 25000,   # 渋谷・新宿・品川・六本木
    "B": 18000,   # 池袋・五反田・秋葉原・恵比寿
    "C": 13000,   # 錦糸町・大崎・中野・立川
    "D": 9000,    # 郊外ターミナル
}
```

> 正確な賃料データは不要。エリア間の相対的なコスト差がわかれば探索の目的は果たせる。
> 将来的に不動産APIと連携する場合は、このテーブルを API 呼び出しに差し替えるだけで対応可能。

## 2. 郵便番号→最寄駅マッピング

### 目的

社員データの入力負荷を下げる。最寄駅がわからなくても郵便番号だけで入力できるようにする。

### エンドポイント

既存の `/parse-csv` を拡張。CSVに `postal_code` 列があれば自動変換:

```csv
id,name,group,postal_code,days_per_week
emp_001,田中太郎,営業部,150-0001,3
```

→ `150-0001` を最寄駅 `shibuya` に自動変換してパイプラインに渡す。

### データソース

国土数値情報の「鉄道データ」+「位置参照情報」（共に無料・オープンデータ）から、
郵便番号の代表座標と駅座標の最短距離で最寄駅を決定。

```
cest/src/cest/data/
├── area_rent.py           # エリア賃料相場
└── postal_station_map.json # 郵便番号→最寄駅マッピング（ビルド時に生成）
```

> マッピングファイルは事前生成してリポジトリに含める（ランタイムで外部API呼び出しはしない）。
> 東京圏（1都3県）で約4,000件。JSONで ~200KB。

## 3. Bedrock 統合

### エンドポイント

```
POST /ask
```

### リクエスト

```json
{
  "question": "営業部への影響を説明してください",
  "context_type": "explore",
  "context_id": "latest"
}
```

> `context_type` は `"explore"` または `"evaluate"`。
> `context_id` は直近の計算結果を参照する識別子。Lambda のメモリ内に直近結果をキャッシュする（永続化不要）。

### 処理フロー

```
1. 直近の計算結果（explore_results または all_combinations）を取得
2. システムプロンプト + 計算結果 + ユーザー質問を Bedrock に送信
3. Bedrock のレスポンスを返却
```

### システムプロンプト

```
あなたはオフィス配置の分析アシスタントです。
CESTの計算結果を基に、質問に回答してください。

ルール:
- 計算結果に含まれるデータのみを使ってください
- 必ず具体的な数字を引用してください
- データにない推測はしないでください
- 「〇〇すべき」という断定は避け、判断材料を提示してください
- 日本語で回答してください
```

### レスポンス

```json
{
  "answer": "営業部（30名）の平均通勤時間は、現オフィス（丸の内）の50分から新宿案では42分に短縮されます（-8分）。ただしp95は68分と比較的高く、千葉方面の社員3名は通勤が80分を超えます。",
  "model": "anthropic.claude-3-haiku-20240307-v1:0",
  "usage": {
    "input_tokens": 1520,
    "output_tokens": 180
  }
}
```

### PDF レポートへの AI サマリ統合

v0.3.2 の PDF レポート生成時に、Bedrock でサマリ文を自動生成して冒頭に挿入:

```
POST /ask
{
  "question": "この分析結果を経営会議向けに3行で要約してください",
  "context_type": "evaluate",
  "context_id": "latest"
}
```

> Bedrock が利用できない場合（API エラー等）は、v0.3.2 のルールベーステンプレートにフォールバック。

| 項目 | 単価 | デモ想定 | 月額 |
|------|------|---------|------|
| Claude 3 Haiku input | $0.25/100万トークン | 1回あたり ~2000トークン | - |
| Claude 3 Haiku output | $1.25/100万トークン | 1回あたり ~200トークン | - |
| 月100回利用 | - | - | **約7円** |

## 4. AWS 構成の変更

v0.3.2 からの変更点:

| 項目 | 変更内容 |
|------|---------|
| Lambda メモリ | 512MB → 1024MB（探索モードの計算量に対応） |
| Lambda タイムアウト | 30秒 → 60秒 |
| Lambda 依存追加 | `boto3`（Bedrock 呼び出し用。Lambda に標準搭載） |
| API Gateway | `/explore` `/ask` エンドポイント追加 |
| IAM Role | `bedrock:InvokeModel` 権限を Lambda の実行ロールに追加 |
| Bedrock モデルアクセス | `anthropic.claude-3-haiku-*` を有効化（Bedrock コンソールで1クリック） |

> S3、CloudFront、CI/CD パイプラインは変更なし。

## 5. Frontend 変更

### トップタブの追加

v0.3.2 のタブ構成「データ入力 | 結果」に「探索」タブを追加:

```
データ入力 | 探索 | 結果
```

- 「探索」は新規タブ。「結果」は v0.3.2 のまま（感度分析タブは v0.3.2 で廃止済み）
- データ入力で社員データを読み込み → 探索タブで最適エリアを見る → 気になるエリアを候補にして結果タブで比較、の導線

### 探索タブの構成

```
┌─────────────────────────────────────────────────┐
│ 探索条件                                         │
│  坪単価上限: [____] 円/月   必要面積: [____] 坪  │
│  [探索実行]                                      │
├─────────────────────────────────────────────────┤
│ 現状診断（baseline_diagnosis があれば表示）        │
│  現オフィス: 丸の内  平均通勤52分  90分超6人      │
│  ⚠ 収容率が100%を超えています                    │
├─────────────────────────────────────────────────┤
│ 探索結果マップ                                   │
│  ┌────────────────────────────────────┐          │
│  │  地図上に候補エリアをプロット      │          │
│  │  パレート最適=青、それ以外=グレー  │          │
│  │  クリックで詳細表示               │          │
│  └────────────────────────────────────┘          │
├─────────────────────────────────────────────────┤
│ 探索結果テーブル                                 │
│  順位 | 駅名 | ランク | 推定賃料 | 平均通勤 |    │
│       |      |        |          | p95     |    │
│       |      |        |          | 60分圏% |    │
│  パレート最適をハイライト                        │
│  vs_baseline があれば「現状比」列も表示           │
└─────────────────────────────────────────────────┘
```

使うフィールド（`/explore` レスポンス）:
- `explore_results[]`: `station_name`, `area_rank`, `estimated_rent_total`, `avg_trip_minutes`, `p95_trip_minutes`, `within_60min_pct`, `over_90min_count`, `is_pareto_optimal`, `vs_baseline`, `department_summary`
- `pareto_station_ids[]`
- `baseline_diagnosis`（あれば）
- `metadata`: `total_stations_evaluated`, `after_budget_filter`, `pareto_optimal_count`, `computation_time_ms`

### 探索結果から比較モードへの連携

探索結果テーブルの各行に「候補に追加」ボタンを設置:
- クリックすると、その駅をオフィス候補として `/evaluate` のリクエストに追加
- 追加後「結果タブで比較する」ボタンで結果タブに遷移

> これにより「探索で見つけたエリアを、比較モードで詳細分析」という導線が繋がる。
> 探索結果の駅を候補に変換する際、賃料はエリア相場テーブルの値をデフォルトとして使う。

### AI チャット UI

結果タブ・探索タブの下部に、折りたたみ式のチャットエリアを設置:

```
┌─────────────────────────────────────────────────┐
│ 💬 AI に質問する                          [▼]   │
├─────────────────────────────────────────────────┤
│ （展開時）                                       │
│ [質問を入力...                        ] [送信]  │
│                                                  │
│ AI: 営業部（30名）の平均通勤時間は、現オフィス   │
│     の50分から新宿案では42分に短縮されます。      │
│                                                  │
│ 質問例:                                          │
│  ・営業部への影響は？                            │
│  ・一番コスパが良い案は？                        │
│  ・経営会議向けに要約して                        │
└─────────────────────────────────────────────────┘
```

- `POST /ask` に `question` と `context_type`（`"explore"` or `"evaluate"`）を送信
- レスポンスの `answer` を表示
- ローディング中は「分析中…」と表示
- エラー時は「AI分析は現在利用できません」と表示し、チャットエリアをグレーアウト
- 会話履歴は持たない（1問1答）

### 郵便番号対応

データ入力タブの CSV 説明文を更新:
- 必須列の説明に `postal_code` を追加（`station_id` の代わりに使える旨を記載）
- 「郵便番号を入力すると最寄駅に自動変換されます」の説明

### PDF レポートの AI サマリ

v0.3.2 の PDF レポート機能がある前提で:
- PDF 生成前に `/ask` で「経営会議向けに3行で要約」をリクエスト
- 成功すれば PDF 冒頭に AI サマリを挿入
- 失敗すればサマリなしで PDF 生成（v0.3.2 のルールベース部分だけ）

## 実装しないもの

| 項目 | 除外理由 |
|------|----------|
| 実経路API連携（NAVITIME等） | コストが高い。推定方式で目的は果たせる。将来的に差し替え可能な設計にしておく |
| 不動産API連携 | 日本の商業不動産APIは非公開/高額。エリア相場テーブルで代替 |
| 社員セルフ入力フォーム（Cognito） | v0.4 のスコープ外。郵便番号マッピングで入力負荷は十分下がる |
| Step Functions | 単一 Lambda で計算可能。オーバーエンジニアリングを避ける |
| DynamoDB（結果永続化） | Lambda メモリ内キャッシュで十分。就活デモに永続化は不要 |
| 会話履歴の保持 | /ask は1問1答。チャット履歴は持たない（コスト・複雑性の観点） |
