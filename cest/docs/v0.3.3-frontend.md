# CEST v0.3.3 Frontend

**担当**: pachi
**前提**: v0.3.3 backend がマージ済みであること
**ステータス**: 未着手

---

## 背景

v0.3.2 リリース後のドッグフーディングで、UI と意思決定支援の観点で課題が見つかった。
v0.3.3 では **フロントのみ** の改修で対応する（バックエンドの変更は既にマージ済み想定）。

### バックエンド側で既に対応されていること（フロントから利用するだけ）

- パレート判定の主軸が「ワースト5%通勤」から「平均通勤」に変わっている
- 「希望定員（採用余地）」を制約として送れるようになっている
- 「無駄拠点を含む案は除外」する処理がある（固定オフィスは例外）
- 各案に「固定オフィスのうち使われなかったもの」の情報がついている
- 制約絞り込みステップの集計に「無駄拠点フィルタ後の件数」が追加されている

フロント担当が触るのは `web/index.html` のみ。バックエンドコードに変更は不要。

### v0.3.2 → v0.3.3 で増えた / 変わった API フィールド

**リクエスト（フロントから送る）の追加フィールド**:

```json
{
  "settings": {
    "min_total_capacity": 320,    // 希望定員（人）。空欄なら送らない

    ],
    "group_together": [
      ["セールス", "コーポレート"]   // 同じセット番号の部署をグループ化

```

**レスポンス（バックエンドから返ってくる）の変更点**:

## 解決したい課題

1. **散布図の軸と意思決定の軸が乖離している** — 散布図のX軸と裏で動いている判定軸が違うと、画面上で「通勤が改善されてる」のに灰色（劣位）の案が出てユーザーが混乱する
2. **「p95」という用語がわかりにくい** — 経営層や採用担当が見たときに意味が伝わらない
3. **候補ゼロになったときに原因がわからない** — 制約を厳しくしすぎたとき、どのフィルタで全滅したかが見えず、ユーザーが詰む
4. **オフィスCSVの列名・値が厳密すぎる** — 業務担当が用意するCSVは列名や値表記が多様
5. **「ラストマイル」「1人あたり面積」など専門用語が入力UIに残っている** — 経営層が触る前提なら直感的な言葉に
6. **拠点数の選択UIが分かりにくい** — 「1箇所/2箇所/3箇所」チップ選択より「最大何拠点まで検討するか」のほうが直感的
7. **部署別の固定・グループ化ができない** — バックエンドはサポートしているが UI がない
8. **マップに採用案のオフィスしか表示されない** — 候補オフィス全体や現状オフィスとの位置関係が見えない
9. **サンプルデータの設計が悪く、パレート最適案が縦一列に並ぶ** — 1拠点最強の設計になっていて k=2 案が成立しない
10. **本番デプロイのたびに API のURLを書き換える必要がある** — ローカル開発と本番でコードが分岐するのは保守上よくない

## 対応してほしいこと

### 必須

#### A. パレート判定軸の変更と用語整理

**A-1. 散布図の主軸を「平均通勤」に変更**

- 散布図2枚（賃料系2軸 × 通勤）のX軸を「平均通勤」に
- 軸ラベル・凡例・タイトル・ツールチップの表記を「ワースト5%」から「平均通勤」に
- ワースト5%は補助情報として残してOK（案カードや部署別テーブル等）

**A-2. 散布図のパレートフロンティア破線を削除**

- 旧仕様では Pareto 点同士を破線でつないでいた
- 3軸判定では2D平面上の単調曲線にならないため意味がない
- 破線描画と凡例の該当エントリを削除
- 代わりに凡例に「現状オフィス」のアイコン説明を追加（J で詳述）

**A-3. 「p95」という用語の置き換え**

UI 表記を以下に統一:

| 旧 | 新 |
|---|---|
| 通勤 p95 / p95通勤 | 通勤ワースト5% |
| 最大通勤時間 p95 | 通勤ワースト5%の上限 |
| 散布図ラベル「通勤 p95（分）」 | 「平均通勤（分）」 |

該当箇所: 散布図ラベル / 凡例 / ツールチップ / 現状診断パネル / 案カードのKPI / 部署別テーブル / 候補タブ

**A-4. 平均通勤の上限を入力欄の上位に**

- 「平均通勤の上限」を予算欄の直下に配置
- 「通勤ワースト5%の上限（外れ値を抑える）」は詳細設定（折りたたみ）の中へ移動

理由: 経営判断では平均通勤のほうが直感的な指標。ワースト5%は外れ値対応の補助制約という位置づけに。

---

#### B. 候補ゼロ時の自動診断パネル

候補が1件もないときに、結果セクションの代わりに専用パネルを表示する。

UI イメージ:
```
⚠ 該当する案が見つかりません

制約フィルタを通った件数:
  全21 → 容量0 → 予算0 → 通勤0 → 無駄なし0

考えられる原因:
  ・収容フィルタで全21案が落ちました
    → 全員入るオフィス（または2拠点組み合わせ）がない
    → 容量を増やすか、拠点候補を増やしてください
  ・駅不明オフィス2件: 渋谷新本社、品川支社
    → これらは通勤計算ができず案から外れます

サイドバーの制約値を緩めて再計算してください
```

判定: 制約絞り込みの集計を見て、どの段階で件数が 0 になったかを判別し、対応する原因メッセージを出す。

候補が1件以上ある場合はこのパネルは非表示。

---

#### C. オフィスCSVの読み込みを柔軟に

**C-1. 列名のエイリアス対応**

業務担当が用意するCSVは列名が一定しないため、以下を吸収する:

| 認識する列名（いずれか） | 対応するフィールド |
|---|---|
| `name` / `オフィス名` / `拠点名` / `名称` / `名前` | name |
| `nearest_station_id` / `station` / `最寄駅` / `駅` / `最寄り駅` / `最寄` / `駅名` / `駅id` | nearest_station_id |
| `last_mile_minutes` / `徒歩` / `徒歩(分)` / `徒歩分` / `last_mile` / `駅徒歩` / `駅から徒歩` | last_mile_minutes |
| `rent_jpy_month` / `rent` / `賃料` / `家賃` / `月額賃料` / `月額家賃` | rent_jpy_month |
| `capacity` / `capacity_people` / `収容` / `収容人数` / `定員` / `人数` | capacity_people |

列の順序にも依存しないこと。

**C-2. 値の表記ゆれを吸収**

- 賃料: `8000000` / `800万` / `8,000,000` / `8,000,000円` / `￥8000000` をすべて受け入れる
- 徒歩: `5分` / `5 分` / `5` を受け入れる
- 駅: 半角英数字なら駅ID扱い、日本語なら `station_master.json` の `name` から逆引き（`渋谷駅` のような末尾「駅」も許容）
- 収容: `100人` / `100,000` の `人` `,` を除去

**C-3. 駅が解決できなかった行を通知**

CSV 読み込み完了後、駅マスタにマッチしない駅が含まれていた場合、ダイアログで「読み込みは完了しましたが、駅が見つからない行があります: XX, YY ...」と通知。

**C-4. オフィスリストで未設定項目を可視化**

サイドバーのオフィス候補リスト各行で:
- 賃料未設定 → 「⚠賃料未設定」（赤字）
- 定員未設定 → 「⚠定員未設定」（赤字）
- 最寄駅が駅マスタに存在しない → 「⚠駅不明「●●」→このまま計算すると候補から外れます」（赤字 + 行全体を薄赤背景）

#### D. 部署別ルール入力UI

部署を特定オフィスに固定したり、複数部署を必ず同じ拠点に置く制約を、UI から指定できるようにする。

**D-1. サイドバーに「部署別ルール」セクションを追加**

折りたたみセクション（デフォルト展開）として、現在のオフィス（baseline）セクションの下に配置。

説明文:
> 特定部署を必ず特定オフィスに置きたい / 部署同士を必ず同じ拠点にしたい場合に指定
>
> - 固定オフィス: その部署を必ずそのオフィスに配置（指定したオフィスを含まない案は除外）
> - セット: 同じ番号の部署は必ず同じ拠点に配置。例: 営業=1, マーケ=1 なら2つは必ず同じオフィス

**D-2. 部署一覧の動的描画**

社員データから抽出した部署一覧を行として表示。各行は3列：

| 列 | 内容 |
|---|---|
| 部署名 | 部署名（人数を補足表示） |
| 固定オフィス | ドロップダウン（「指定なし」+ 現在のオフィス候補名一覧） |
| セット | 数値入力（空欄=指定なし、1〜9の範囲） |

社員データが空のときは「社員データを読み込むと部署が出ます」と案内。
オフィス候補を追加・削除したらドロップダウンを再描画。

**D-3. 設定の送信**

計算実行時に以下に変換:
- 各部署で「固定オフィス」が選択されていたら、`settings.fixed_assignment` に `{ group, office_id }` の形で追加
- 同じセット番号の部署が2つ以上あれば、`settings.group_together` にグループ配列として追加（単独セット番号は無視）

#### I. マップに全候補オフィスを表示

現在は採用された案のオフィスしかマップに出ない。以下を追加:

- **採用オフィス**: ★（色付き、大きめ、強調）— 既存

#### K. API_BASE をホスト名で自動判定

`web/index.html` で API のURLをハードコードしている箇所を、ブラウザのホスト名で自動判定するようにする。

判定方針:
- `localhost` または `127.0.0.1` で開いている → ローカルAPI（`http://localhost:8000`）
- それ以外 → 本番API Gateway（URL は別途共有）

ローカル開発時は `http://localhost:5500/index.html` で開けばローカルAPIを、本番（CloudFront経由）では API Gateway を自動で叩く。コード1箇所の置き換えで両対応。

#### L. サンプルデータの全面刷新

現在の `SAMPLE_CONFIGS`（IT / メーカー / 外資）は以下の問題があるため再設計する:

- 1社100人規模で k=1 単独で全員入る → 2拠点案が「片方未使用」で除外される
- 社員プールが1つの大きな配列で、全部署が同じ最寄り（新宿等）に集中する → パレートが縦一列に並ぶ
- 候補オフィスに「ゴミ拠点」（誰の最寄りでもない郊外）が混じり、Pareto に乗らない案が増える

**新サンプルの方針**:
- 社員数を **300人** に拡大（中規模企業を想定、複数拠点が現実的な検討対象になる）
- **部署ごとに居住エリアを偏在** させる（実情に近い設計）
  - 例（IT企業）: エンジニアは中央線沿線、プロダクトは千葉系、セールスは神奈川系、コーポレートは埼玉系
- **オフィス候補を都心集中** にする（誰かの最寄りになる立地のみ）
  - 例（IT企業）: 新宿 / 渋谷 / 東京 / 品川 / 池袋 / 新橋
- 各オフィスの容量を 60〜200 人程度に絞る（単独で全員収容できないサイズにして 2拠点案を必然化）
- 制約値（予算・平均通勤上限・希望定員）と現状オフィス（baseline）もサンプルに含める

**サンプル選択時の挙動**:
- 社員データ・オフィス候補に加え、サイドバーの各種制約フィールド・baseline フィールド・部署別ルール（リセット）に値を流し込む
- baseline セクションは開いた状態にする
- API 接続済みなら自動で再計算する

具体値（駅ID・賃料・容量・部署人数・pool 等）は **`v0.3.3-sample-data.md`** に整理して別途共有する。
仕様書側ではフォーマットと意図のみ規定する。

`SAMPLE_CONFIGS` の型を以下に拡張（既存型の置き換え）:

```js
{
  it:     { label, offices[], departments[], pool[], officeDays, numOffices,
            constraints: { maxAvg, budgetMan, desiredCapacity },
            baseline:    { name, station, lastMile, rentMan, capacity } },
  maker:  { /* 同型 */ },
  gaishi: { /* 同型 */ },
}
```

- `departments[]` の各要素に `pool: [駅ID配列]` を持たせる（部署ごとに居住エリア指定）
- 最上位の `pool` は後方互換用（部署別 pool があれば未使用）
- 社員生成ロジックは「部署ごとに `dept.pool` を優先、なければ defaultPool を使う」に拡張

2. **手動で予算を 500 万円に変更して再計算** → 候補ゼロ → 原因表示パネルに「予算フィルタで全滅」が出る
3. **オフィスCSVを「name,駅,賃料,定員」のような日本語列名で読み込む** → 駅・賃料・定員すべて認識される
4. **不正な駅名でCSV読み込み** → 赤背景で警告 → 計算ボタン押下時に確認ダイアログ
5. **部署別ルールでエンジニアリングを新宿固定 + セールスとコーポレートをセット1** → 該当する案だけがパレートに残る
6. **本番URL（CloudFront経由）でアクセス** → ローカルAPIを叩かず、API Gateway 経由で動作
